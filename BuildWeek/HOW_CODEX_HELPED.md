# How Codex Helped Prepare YouNew

## Evidence-safe account

The repository contains implementation and reports consistent with the documented
Codex-assisted workflow. That is the strongest claim supported by the available
artifacts: repository history does not attribute every source line or commit to a
specific Codex session.

The documented collaboration is human-directed. The owner defines the product
problem, audience, priorities, visual direction, acceptance constraints, and
release boundaries. Codex supports the engineering loop through repository
inspection, implementation, focused refactoring, debugging, test execution,
technical auditing, and evidence preparation. The owner reviews the results and
retains every product, rights, distribution, and submission decision.

## Where the workflow is most visible

### 1. Premium image pipeline

The project consolidates image behavior into shared, role-aware components with
focal-point policy, local and remote candidates, deterministic fallbacks,
downsampling, bounded caches, request coalescing, HTTP validation, and
accessibility labels. The documented Codex-assisted workflow is consistent with
implementing and auditing this shared system across image-heavy surfaces.

### 2. Interactive Netherlands map

YouNew contains a custom SwiftUI map with geometry for all twelve provinces,
labels, markers, selection, zoom, pan, typed navigation, and deterministic
hit-testing contracts. The collaboration record is consistent with iterative map
implementation, navigation debugging, geometry tests, and targeted stabilization.

### 3. Governed content and import platform

The repository includes schemas, lifecycle metadata, release definitions,
migrations, validation rules, deterministic import tooling, a bundled runtime
payload, and app-facing repository and index layers. Codex-supported work is
documented around validator hardening, canonical identifiers, import behavior,
runtime integration, and evidence reporting.

### 4. The five-city release

`cities-v0.1.0` brings Amsterdam, Rotterdam, Den Haag, Utrecht, and Eindhoven
through one governed release path. Repository evidence covers canonical IDs,
migration mappings, structural import checks, app consumers, and focused test
contracts without claiming equal content depth for every city.

### 5. QA and Build Week packaging

The project includes build, unit, UI, accessibility, routing, media, content,
privacy, data, import, and release checks. The documented workflow uses those
checks to surface failures and preserve evidence boundaries rather than convert a
partial result into an “all green” claim. Build Week preparation then turns the
verified product scope into a focused demo, technical narrative, limitations, and
owner handoff.

## What Codex contributed to the working method

- Fast navigation across a large SwiftUI and data repository.
- Consistent movement from requirement to implementation to evidence.
- Targeted stabilization of existing systems instead of unnecessary expansion.
- Explicit separation of source-level capability, targeted verification, and
  production-readiness claims.
- Documentation that makes unresolved risks visible to the owner and judges.

## Attribution boundary

YouNew does not claim that Codex independently designed the product, authored a
particular file, or made final decisions. Stronger authorship or session-specific
claims would require an owner-approved session export or task record. It also does
not claim that the in-app assistant is Codex, ChatGPT, a live OpenAI service, or
GPT-5.6. The demonstrated assistant is local and deterministic.
